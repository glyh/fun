# Wiring Checklist: Features needed before connecting to old pipeline

## Done

- [x] `let rec` / `Fix` in surface syntax and core evaluation
- [x] Decision-tree match compilation and exhaustiveness checking for nominal ADT constructor patterns
- [x] Literal patterns for `I64`, `Bool`, and `Unit`

## Blocking

- [x] Char literal patterns ‚Äî single-quoted `Char` literals in expressions and match branches
- [x] Tuple patterns ‚Äî destructuring `(a, b, c)` in match branches
- [x] Or-patterns ‚Äî `A | B -> ...` combining alternatives
- [x] Record patterns ‚Äî `Foo {x; y}` or `Foo {x; _}` destructuring ([plan](4.core_tt_records.md))

## Probably blocking

- [x] Polymorphic `==`/`!=` ‚Äî specialize on implicit type params for now, with monomorphic equality primitives underneath ([plan](5.core_tt_type_specialized_equality.md))
- [x] Parameterized struct record types ‚Äî modeled as functions returning structs, with implicit insertion for record construction and `Pair.field` access
- [x] Qualified constructor/type paths in patterns ‚Äî expression/type access uses struct field access; patterns support path syntax and resolution for `M.Red`, nested paths, aliases like `N.Red`, and record patterns like `M.Point {x; y}` ([plan](6.core_tt_qualified_paths.md))
- [x] Struct variant syntax parity ‚Äî intentionally not carried forward: structs are structural, ADTs are nominal; translate old `let Color = struct | Red | Green end` cases to core_tt `type Color = Red | Green in ...`
- [x] Recursive parameterized nominal ADTs ‚Äî support recursive nominal references in constructor payloads, e.g. `type List a = Cons (a * List a) | Nil in ...`
- [x] `method`/`self`/`Self` in struct record types ‚Äî methods inside `struct ... end` lower to receiver functions; `self` names the method receiver, and `Self` names the enclosing record type after field declarations
- [ ] Translate main semantic/runtime regression tests to core_tt equivalents ‚Äî constants/lets/recursion/matches/records mostly translate, but no shared test corpus exists yet

## Lower priority

- [ ] Record-type-directed field inference ‚Äî partial-struct unification covers some field access cases, but not old-pipeline named-record inference
- [ ] Main-pipeline record declaration syntax compatibility ‚Äî old pipeline accepts `type point = {x: I64; y: I64}`; core_tt intentionally models record types as `struct` values
- [ ] Top-level file/import pipeline for core_tt ‚Äî old pipeline tests import resolution/circular import detection; core_tt is still expression/test-parser only
- [ ] Main parser syntax parity ‚Äî old parser tests cover comments, `;;`, function-binding sugar, import syntax, and type-variable syntax that core_tt's test parser does not currently target
