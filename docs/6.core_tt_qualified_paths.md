# Qualified paths in core_tt

Qualified access uses ordinary struct field access in expression and type positions, and path-aware resolution in pattern positions.

## Expressions and types

Expression-side qualified names are parsed as chained field access:

```fun
M.Red
A.B.X 7
M.Point
```

Public types and constructors inside structs are exposed as public fields, so expression/type paths do not need a separate qualified-name AST.

## Constructor patterns

Pattern heads preserve their path during surface parsing so elaboration can validate the path against the scrutinee type:

```fun
match c with
| M.Red -> 1
| M.Green -> 2
end
```

Nested paths and aliases work by resolving through public struct fields:

```fun
let N = M in
match M.Red with
| N.Red -> 1
end
```

After validation, runtime matching still uses the resolved nominal constructor identity rather than string paths.

## Record patterns

Record patterns also support qualified type paths:

```fun
match p with
| M.Point {x; y} -> x + y
end
```

Elaboration resolves `M.Point`, checks it is the same structural record type as the scrutinee, and then elaborates the field subpatterns normally.

## Non-goals

- Expression-side paths remain ordinary field access.
- File/package import path semantics are handled by `Core_loader`, not by qualified-name resolution.
- Runtime constructor matching should stay nominal-id based rather than path-string based.
