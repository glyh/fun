# Qualified constructor/type paths in core_tt

## Goal

Bring core_tt closer to old-pipeline path syntax for constructors and type names in expressions and patterns.

The old pipeline supports forms like:

```fun
M.Red
A.B.X(n)
N.Red
```

where `N` may be an alias for another struct/module. In core_tt, expression-side dotted access mostly already works through ordinary struct field access, but patterns still rely on unqualified constructor/type names.

## Current state

Expression and type positions already use general field access:

```fun
S.Red
S.Color
A.B.X 7
```

These parse as chained `FieldAccess` expressions and work when each segment is a public struct field. Public types and constructors inside structs are exposed as fields, so there is no separate expression-level qualified-constructor representation today.

Pattern positions are the missing part:

```fun
match c with S.Red -> 1 | S.Green -> 2 end
match x with A.B.X(n) -> n end
match p with M.Point {x; y} -> x + y end
```

Core_tt currently represents constructor patterns and record-pattern type names as plain strings, so it cannot preserve or resolve module paths in patterns.

## Proposed design

Keep expression paths as field access. Add path-aware pattern syntax and elaboration only where the current representation loses information.

Surface patterns should carry an optional path:

```ocaml
PatCon of string list * string * pat list
PatRecord of { typ_path : string list; typ : string; fields : (string * pat option) list; partial : bool }
```

Examples:

```fun
Red        => PatCon ([], "Red", [])
S.Red      => PatCon (["S"], "Red", [])
A.B.X(n)   => PatCon (["A"; "B"], "X", [PatBind "n"])
M.Point {x} => PatRecord { typ_path = ["M"]; typ = "Point"; ... }
```

The core pattern representation can remain mostly unchanged if elaboration resolves qualified surface patterns to the same unqualified constructor and record-pattern core forms after validating that the path points at the expected public constructor/type field.

## Implementation steps

1. Extend surface pattern representation.
   - Update `lib/semantic/core/surface.ml` so `PatCon` stores `path * name` instead of only `name`.
   - Add `typ_path` to `PatRecord`.
   - Update all existing pattern construction sites and tests for the new constructors.

2. Parse dotted pattern heads.
   - Add a small dotted identifier helper to `lib/semantic/core/core_parser.mly`.
   - Parse:
     - `Red`
     - `S.Red`
     - `A.B.X(p)`
     - `M.Point {x; y}`
   - Preserve existing lowercase binders and `_` behavior.
   - Keep expression parsing unchanged; expression `S.Red` should continue to be ordinary `FieldAccess`.

3. Add path resolution for pattern heads.
   - In `lib/semantic/core/elaborate.ml`, add a helper that resolves a path from the elaboration context by walking public `VStruct` fields.
   - For `S.Red`, resolve `S`, then field `Red`.
   - For aliases like `let N = S in N.Red`, this should work naturally because `N` evaluates to the same struct value shape.
   - Reject private fields and unknown path segments with existing elaboration errors where possible.

4. Elaborate qualified constructor patterns.
   - For unqualified `Red`, keep the current behavior: resolve by comparing against the known scrutinee nominal type's constructor list.
   - For qualified `S.Red`, first resolve the path expression to a constructor value, then verify that it belongs to the scrutinee nominal type.
   - Preserve existing arity checks, binder checks, or-pattern behavior, match compilation, and exhaustiveness behavior.
   - Do not add path information to runtime constructor matching unless validation proves the existing nominal-id check is insufficient.

5. Elaborate qualified record patterns.
   - Resolve `M.Point` to a public type field and require it to be the same `VStruct` type as the scrutinee.
   - Then reuse existing record-pattern field elaboration.
   - Support aliases like `let N = M in match p with N.Point {x; y} -> ... end` if the path resolves through public struct fields.

6. Add tests.
   - Parser tests for dotted constructor and record patterns.
   - Elaboration tests for:
     - `S.Red` / `S.Green` constructor patterns.
     - nested `A.B.X(n)` constructor patterns.
     - alias `let N = S in match S.Red with N.Red -> ... end`.
     - private constructor/type path rejection.
     - qualified record pattern `M.Point {x; y}`.
   - Runtime tests mirroring the old pipeline examples.

7. Update checklist after validation.
   - Once targeted tests and `dune test` pass, mark the qualified-path checklist item complete or narrow it to any remaining expression-level syntax differences.

## Non-goals

- Do not replace expression-side `FieldAccess` with a separate qualified-name AST unless a concrete ambiguity appears.
- Do not add package/file import path semantics here; this is only about paths through struct/module values already in scope.
- Do not change nominal constructor runtime representation unless qualified validation requires it.
