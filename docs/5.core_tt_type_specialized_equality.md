# core_tt type-specialized polymorphic equality

## Context

The old pipeline has polymorphic `==` and `!=`. core_tt currently treats them as monomorphic `I64 -> I64 -> Bool` primitives.

Full typeclasses/instance resolution are still desirable later, but they are more design-heavy than this migration slice needs. For now, implement polymorphic equality by specializing on an implicit type parameter. This is intentionally a pragmatic bridge: ugly enough to replace later, but still useful because type-parameter specialization is a real feature in dependently typed languages.

Target shape:

```fun
(==) : {A : Type} -> A -> A -> Bool
(!=) : {A : Type} -> A -> A -> Bool
```

User code remains:

```fun
x == y
```

The parser keeps a fixed built-in operator table. Infix `==` elaborates to an ordinary function application of the built-in equality binding, and elaboration inserts the implicit type argument:

```fun
(==) {A} x y
```

where `A` is inferred from the operands. Prefix/operator-as-value use must use the parenthesized operator identifier form, OCaml-style:

```fun
(==) a b
```

Bare prefix operator syntax like `== a b` is intentionally invalid because it creates parsing ambiguity with infix operator grammar.

## Recommended approach

Add monomorphic equality primitives for the primitive value types, then define polymorphic `==` and `!=` by matching on the implicit type argument:

```fun
let (==) : {A : Type} -> A -> A -> Bool =
  fun {A : Type} ->
    match A with
    | I64 -> eq_i64
    | Bool -> eq_bool
    | Char -> eq_char
    | Unit -> eq_unit
    | _ -> panic "unsupported equality type"
    end
```

and similarly for `!=`.

Type matches are open-world, like integer/char literal matches: matching `I64 | Bool | Char | Unit` is not exhaustive. They always need a catch-all unless the match compiler can prove coverage some other way. For this temporary equality bridge, the catch-all uses `panic`. Its result type is inferred from the branch's expected type, so callers should not have to pass the type argument explicitly.

The important point is that `A` is an implicit argument. This preserves old-pipeline call sites while keeping specialization explicit in core_tt.

## Operator syntax scope

This slice should **not** add operator overriding, user-defined custom operators, or operator declarations.

Instead, keep a fixed built-in operator table in the core test parser/elaborator. The table should encode precedence tiers and associativity for the operators core_tt already recognizes or needs for old-pipeline parity:

```text
tier 7, left-assoc: * / %
tier 6, left-assoc: + -
tier 4, left-assoc: == != < <= > >=
```

The symbolic operator remains syntax for applying a fixed binding name. For example:

```fun
x == y
```

parses/elaborates as applying `(==)` to `x` and `y`; it is not a locally overrideable operator lookup.

When an operator is used as a value or in prefix position, it must be parenthesized:

```fun
(==) x y
let same = (==)
```

Bare prefix operators are not valid:

```fun
== x y
```

The parser should reject that form rather than trying to disambiguate it.

This keeps the immediate implementation small while making precedence explicit enough for translating old tests. General operator overloading/custom operators can be revisited with typeclasses or a separate operator declaration design later.

## Implementation steps

1. Add `panic`.
   - Add a temporary primitive:
     ```fun
     panic : {A : Type} -> String -> A
     ```
   - User code should usually call it without an explicit type argument:
     ```fun
     panic "unsupported equality type"
     ```
     Elaboration infers `A` from the expected type.
   - If core_tt does not have `String`, use a simpler internal primitive with a fixed diagnostic string or add only the parser/runtime support needed for string literals in this context.
   - Evaluating `panic` raises an `EvalError`.
   - This gives open-world type matches a typed catch-all branch.

2. Add monomorphic primitive equality operations.
   - In core primitive typing/reduction, add:
     - `eq_i64`, `neq_i64`,
     - `eq_bool`, `neq_bool`,
     - `eq_char`, `neq_char`,
     - `eq_unit`, `neq_unit`.
   - Keep arithmetic/comparison primitives unchanged.
   - Existing `==`/`!=` can be temporarily retained until the specialized definitions replace them.

3. Add surface/core syntax for implicit lambda binders if needed.
   - core_tt already has implicit Pi/application, but surface lambda syntax may need:
     ```fun
     fun {A : Type} -> body
     ```
   - This should elaborate to an implicit lambda whose type checks against:
     ```fun
     {A : Type} -> ...
     ```

4. Add type patterns.
   - Extend surface/core patterns with primitive type-head patterns, distinct from value literal patterns:
     ```ocaml
     PatType of atom_ty
     CPatType of atom_ty
     ```
   - Parse these when matching on `Type` values:
     ```fun
     match A with I64 -> ... | Bool -> ... end
     ```
   - Keep value patterns like `true`, `1`, and `'a'` separate.

5. Support matching on `Type` values.
   - Extend match-domain classification with a primitive type domain, but treat it as open-ended/non-finite for exhaustiveness.
   - `I64`, `Bool`, `Char`, and `Unit` cases are useful specializations, not a closed universe.
   - A match over `Type` without `_` should be non-exhaustive.
   - Nominal/user type heads can be considered later if specialization needs to grow.

6. Refine the matched type parameter inside branches.
   - For:
     ```fun
     fun {A : Type} ->
       match A with
       | I64 -> eq_i64
       | Bool -> eq_bool
       end
     ```
   - The `I64` branch must typecheck with `A` refined/unified to `I64`, so:
     ```fun
     eq_i64 : I64 -> I64 -> Bool
     ```
     can satisfy the expected branch type:
     ```fun
     A -> A -> Bool
     ```
   - This branch refinement is the key nontrivial piece.

7. Define polymorphic `==` and `!=` in the initial context.
   - Prefer defining them as ordinary core terms over making them elaborator magic.
   - The resulting runtime behavior is still ordinary function application plus type-level specialization.

8. Make fixed operator tiering explicit in the core parser.
   - Do not add custom operator declarations or overriding.
   - Replace the current flat `OP` binop parsing with fixed tiers for built-in operators:
     ```text
     tier 7, left-assoc: * / %
     tier 6, left-assoc: + -
     tier 4, left-assoc: == != < <= > >=
     ```
   - Each operator still parses as application of its fixed built-in binding name.
   - Once `==` has a leading implicit `{A : Type}`, existing implicit insertion should infer `A` from the first operand and check the second operand against the same type.

9. Add focused tests.
   - Passing primitive equality:
     ```fun
     1 == 1
     1 != 2
     true == false
     'a' == 'a'
     () == ()
     ```
   - Polymorphic helper:
     ```fun
     let same : {A : Type} -> A -> A -> Bool =
       fun {A : Type} -> (==) {A}
     in same 1 1
     ```
   - Mismatch rejection:
     ```fun
     1 == true
     ```
   - Unsupported nominal equality rejection for now:
     ```fun
     type Option a = Some a | None in Some 1 == Some 1
     ```

## Fallback option

If type-pattern matching plus branch refinement turns out too large for one slice, implement `==`/`!=` as elaborator built-ins that specialize directly by the inferred operand type.

That fallback should be treated as temporary. The preferred path is real type matching because it introduces the reusable specialization feature.

## Scope notes

- Do not implement full typeclasses or instance search in this slice.
- Do not add equality for user-defined nominal or record types yet.
- Do not add operator overriding, custom operators, or operator declarations yet.
- Keep operator precedence fixed and built-in for now.
- Keep type-pattern exhaustiveness open-world: primitive type heads are not a complete set, and `_` is required for total matches over `Type`.
- Make the replacement easy to remove when typeclasses/instances are introduced later.
