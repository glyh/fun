# Regression coverage plan

## Goal

Track regression tests for current behavior before larger prototype work continues. This plan prioritizes features that already exist and should remain stable through type-case, protocol/typeclass, effects, macro, and CLR/C# rewrite exploration.

## Phase 1: Imports + modules integration

Highest-value gap: local feature tests are strong, but imported modules need more cross-feature regression coverage.

- [ ] imported record type construction and field access
- [ ] imported record pattern matching
- [ ] imported method using `self`
- [ ] imported method using `Self`
- [ ] imported qualified nested constructor pattern
- [ ] imported module alias used in a pattern
- [ ] imported public effect used in a handler
- [ ] imported private constructor remains inaccessible
- [ ] imported private record/module member remains inaccessible through aliases

## Phase 2: Effects regression hardening

Effects are changing fastest, so keep behavior locked down with focused tests.

- [ ] `resume` outside an effect branch is rejected
- [ ] `resume` without an argument is rejected at parse or elaboration time
- [ ] continuation reuse raises a clear runtime error, if surface-expressible
- [ ] full multi-operation handler removes the handled effect
- [ ] partial multi-operation handler remains effectful
- [ ] handler branch can perform another effect handled by the same match
- [ ] direct unhandled `perform` raises a clear top-level runtime error
- [ ] tuple operation payload patterns work statically and at runtime
- [ ] record operation payload patterns work statically and at runtime

## Phase 3: Type-case / equality baseline

Before expanding type-case, lock current behavior around primitive type-head matching and equality.

- [ ] primitive type-head matching works for `I64`
- [ ] primitive type-head matching works for `Bool`
- [ ] primitive type-head matching works for `Char`
- [ ] primitive type-head matching works for `Unit`
- [ ] open `Type` matches require a fallback where appropriate
- [ ] `==` accepts same-typed primitive operands
- [ ] `!=` accepts same-typed primitive operands
- [ ] `==` rejects mismatched operand types
- [ ] nominal equality remains unsupported or explicit until protocols/typeclasses define it
- [ ] record equality remains unsupported or explicit until protocols/typeclasses define it

## Phase 4: Records and methods local edge cases

Existing local coverage is already strong. Add only edge cases that protect intended semantics.

- [ ] method returning `Self`
- [ ] method accepting parameterized `Self`
- [ ] private helper used by a public method
- [ ] method on parameterized record with extra explicit argument
- [ ] record pattern with qualified alias through an imported module

## Phase 5: Qualified paths

Local qualified path coverage is solid. Add import-backed path cases to protect module/import interaction.

- [ ] imported nested module constructor pattern
- [ ] alias of imported module used in constructor pattern
- [ ] alias of imported module used in record pattern
- [ ] private imported constructor remains inaccessible
- [ ] wrong-nominal qualified constructor pattern is rejected across imports

## Suggested implementation order

1. Add semantic import integration tests.
2. Add backend/runtime import integration tests for cases that evaluate to concrete values.
3. Add effects hardening tests while effect code is still fresh.
4. Add type-case/equality baseline tests before implementing more type-case behavior.
5. Fill records/methods and qualified-path edge cases only where coverage is still missing.
