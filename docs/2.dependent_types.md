# Dependent Types: Bidirectional Checking + NbE

## Context

Replace the HM constraint-based typechecker with a dependent type system using bidirectional checking and normalization-by-evaluation (NbE). This collapses the type/value boundary and reshapes the entire typechecker. Doing it first means everything else (effects, VM backend) builds on the right foundation.

## Design decisions

- **Variable representation**: de Bruijn indices in syntax, de Bruijn levels in semantic values. All binders are unnamed in the core — names are only needed in the surface language for user-facing lookup.
- **Polymorphism**: Metavariable solving with pattern unification (Miller). Unsolved metas at let-bindings get generalized (Phase 7).
- **Universes**: Type : Type for now (inconsistent but simple). Upgrade path: cumulative `U(i)`.
- **Primitives**: named primitives (`+`, `==`, etc.) live as stuck neutrals (`HPrim`) until fully applied.
- **Metavariables**: `InsertedMeta(id, bds)` tracks which binders are `Bound` (lambda/pi) vs `Defined` (let). When solving, only abstract over `Bound` variables.
- **Elaborator input**: `Surface.t`, decoupled from `Desugared_ast`. Types are expressions.

## Phase 1: Core NbE + bidirectional checking (done)

### Core language

Terms use de Bruijn indices: Var, Lam, Ap, Let, Pi, U, Atom, AtomTy, If,
Prod, ProdTy, Fix, Prim, Meta, InsertedMeta.

### Semantic domain

Values use de Bruijn levels. Three kinds of stuck term, each with a distinct
unification strategy:

- **VRigid(lvl, spine)** — bound variable. Unify by level identity + spine.
- **VFlex(id, spine)** — unsolved meta. Dispatch to pattern-unification solver.
- **VNeutral(ty, head, frames)** — primitive or variable under elimination frames
  (FApp, FIf, FProj, FDot). Decompose head + frames.

### NbE

- `eval` normalizes terms to values; `quote` reads back values to terms.
- `conv` checks extensional equality — under binders, it introduces a fresh
  rigid variable and compares bodies. Eta-equality for lambdas: VLam vs
  non-lambda η-expands the non-lambda side.
- `force` chases solved metas through VFlex.

### Pattern unification

Miller's pattern fragment. `Renaming` maps spine variable levels to solution
lambda levels. `rename` traverses the RHS, replacing spine variables with
de Bruijn indices into the solution lambda chain. Includes occurs check.
`solve` builds the renaming, renames, wraps in lambdas, evals at depth 0.

Errors: NonLinearSpine, NonVariableInSpine, VarNotInSpine, OccursCheck,
CannotUnify, SpineLengthMismatch, NeutralHeadMismatch, FrameMismatch, etc.

### Bidirectional elaboration

`infer` produces (core term, type value); `check` verifies against an
expected type. Ctx module groups context operations (bind, define, lookup,
fresh_meta, eval, quote, unify). Name table uses a balanced tree storing
de Bruijn levels — O(log n) lookup, no O(n) bumping on each binder.

Application inference: known VPi → check argument against domain.
Stuck type (VFlex/VRigid/VNeutral) → create fresh domain/codomain metas,
unify function type with Π(?dom, ?cod).

Built-in types (I64, Bool, Unit, Char, Type) and primitives (+, -, *, /,
%, ==, !=, <, >, <=, >=, not) are pre-loaded in `init_ctx`.

## Phase 2: Structs + tuple projection (done)

**Structs** as the universal construct for records, modules, and namespaces.

- `struct let x = e1; let y = e2 end` — bindings evaluate left-to-right,
  each sees all previous bindings.
- `open M in body` — brings struct fields into scope.
- `e.field` — named field access (Dot).
- `e.0`, `e.1` — positional tuple projection (Proj).

Eval: Struct evaluates bindings sequentially; Open extends env with public
fields. Conv/unify compare field names + values pointwise.

## Phase 3: Struct fields + width subtyping (done)

Three kinds of struct members:

| Kind | Syntax | Visible outside? |
|---|---|---|
| Constructor field | `x: I64;` | yes (in type, required at construction) |
| Computed field | `pub let x = e` | yes (in type, derived) |
| Private binding | `let x = e` | no |

**Width subtyping**: partial structs (marked internally) unify with any struct
containing at least those fields. Used for stuck field access — when the
scrutinee type is unknown, emit a partial struct constraint with a fresh meta
for the field type. Resolves when the struct type becomes concrete.

Structs also accept `type` and `pub type` bindings for nominal ADTs scoped
to the struct.

## Phase 4a: Nullary ADTs (done)

ADTs are nominal, structs are structural — they don't mix.

`type` is a binder like `let`: `type Color = Red | Green | Blue in body`.
Both the type name and all constructors are in scope for the body.

Each `type` definition gets a unique nominal id (`NominalId.fresh()`). Two
separately-defined types with the same name are distinct — unification
compares by id, not name.

Constructors are values (`VCon { name; spine; nominal }`). Nullary
constructors have empty spine. The nominal field links back to the parent type.

`Con of string` is a quote-only artifact (like `Meta`) — eval scans the
environment by name to recover the VCon/VNominal. The elaborator never
produces `Con`; it uses `Var ix` for constructor references.

Inside structs, `TypeBind` carries pre-computed VNominal/VCon values so
the evaluator stores them directly without routing through `eval`.

## Phase 4b: Parameterized ADTs with payloads (done)

Constructors gain type parameters and payload arguments. In the DT setting,
constructors are just Pi-typed lambda chains:

```
type Option a = Some a | None in body

Option : (a : Type) -> Type
Some   : (a : Type) -> a -> Option a
None   : (a : Type) -> Option a
```

`Some` evaluates to a curried VLam chain: applying to `I64` gives
`λx. VCon("Some", [I64, x], Option I64)`. Applying to `42` gives
`VCon("Some", [I64, 42], Option I64)` — a fully saturated constructor
value ready for pattern matching (Phase 5).

All type arguments are explicit in this phase — the user writes `Some I64 42`,
not `Some 42`. Implicit arguments arrive in Phase 6.

### Key design points

- **VNominal** carries `params : value list` and typed constructor entries
  `(name * payload_type option) list`. Unification checks id equality AND
  unifies params element-wise (so `Option I64 ≠ Option Bool`).
- **NomRef** and **Ctor** are term-level representations for nominal type
  references and constructor bodies inside lambda chains. Both use name-based
  env lookup at eval time (same mechanism as `Con`).
- **build_ctor** generates the curried Lam chain (value) and Pi chain (type)
  for each constructor from the type params and optional payload type.
- **Struct TypeBinding** works the same as Phase 4a but with `params = []`
  and `ctors' = [(name, None)]` for nullary constructors inside structs.
- **VCon unification** added for completeness (name + spine equality),
  matching what `conv` already handles.

### Bugs fixed during review

- **Nonlinear spine detection**: `Renaming.of_spine` compared against the
  wrong tuple projection (target level instead of source level), silently
  accepting `?M[x, x]` patterns. Fixed to compare source levels.
- **VNominal param comparison**: both `unify` and `conv` ignored type params,
  making `Option I64` equal to `Option Bool`. Fixed to unify/compare params
  element-wise.

## Phase 5: Pattern matching

Pattern matching on nominal ADTs with exhaustiveness checking and nested
patterns.

```
match scrutinee
| Some(x) -> x
| None -> default
end
```

Eval dispatches on the constructor name of a VCon scrutinee. Sub-patterns
bind spine elements. Wildcard/variable patterns match anything.

Exhaustiveness checking reuses the Maranget-style machinery from
`lib/semantic/match/` (matrix decomposition, missing pattern diagnostics).

## Phase 6: Implicit arguments

Make type arguments implicit via `{a : Type}` Pi syntax. The elaborator
inserts metas or propagates expected types to fill implicit arguments.

```
None : {a : Type} -> Option a
Some : {a : Type} -> a -> Option a

None           (* elaborator inserts meta for {a} *)
None {I64}     (* explicit override *)
```

Pi and Ap gain an explicitness flag (Implicit | Explicit). Three elaboration
cases: explicit `f {arg}`, expected type propagation, and fresh meta insertion.

## Phase 7: Generalization at let-bindings

After elaborating a let-bound value, collect unsolved metas (including
those from Phase 6 implicit arg insertion), abstract over them as Pi-bound
type parameters. Recovers HM-style polymorphism:
`fun x -> x` gets type `(a : Type) -> a -> a`.
