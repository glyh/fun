# Dependent Types: Bidirectional Checking + NbE

## Context

Replace the HM constraint-based typechecker with a dependent type system using bidirectional checking and normalization-by-evaluation (NbE). This is the most disruptive planned change — it collapses the type/value boundary and reshapes the entire typechecker. Doing it first means everything else (effects, VM backend) builds on the right foundation.

## Design decisions

- **Variable representation**: de Bruijn indices in syntax, de Bruijn levels in semantic values. All binders are unnamed — the string field in `term` was removed. Names are only needed in the surface language for user-facing lookup.
- **Polymorphism**: Metavariable solving with pattern unification. Unsolved metas at let-bindings get generalized (not yet implemented — Phase 7).
- **Interpreter**: `eval` (in `nbe.ml`) already evaluates terms to values. A stand-alone REPL interpreter on Core terms just wraps `eval` with primitive implementations and pretty-printing.
- **Universes**: `Type : Type` for now (inconsistent but simple). Upgrade path: replace `U` with `U of int`, cumulative universes. NbE machinery stays the same.
- **Primitives**: `Prim of string` in syntax, compared by name. A table maps names to types and reduction rules. No separate `VPrim` value — primitives are `VNeutral { head = HPrim name }` until applied to enough arguments.
- **Metavariables**: `InsertedMeta of meta_id * bd list` tracks which binders are `Bound` (lambda/pi) vs `Defined` (let). When solving, only abstract over `Bound`.
- **Elaborator input**: The elaborator defines its own input type (`Surface.t`), decoupled from `Desugared_ast`. Types are expressions.

### Term type (core.ml)

All binders are unnamed — binding is purely by de Bruijn index.

```ocaml
type term =
  | Var of ix
  | Lam of term
  | Ap of term * term
  | Let of term * term * term    (* let _ : A = def in body *)
  | Pi of term * term            (* (x : A) -> B *)
  | U                            (* Type : Type *)
  | Atom of Syntax.Ast.Atom.t
  | AtomTy of atom_ty
  | If of term * term * term
  | Prod of term list
  | ProdTy of term list
  | Fix of term
  | Prim of string
  | Meta of meta_id               (* residual meta in quoted output *)
  | InsertedMeta of meta_id * bd list  (* fresh meta with bound/defined mask *)
  | Proj of term * int               (* positional tuple projection: e.0 *)
  | Struct of (string * term) list    (* namespace struct *)
  | Open of term * term               (* open S in body *)
```

### Value type (core.ml)

Three stuck constructors with distinct unification strategies:

- `VRigid { lvl; spine }` — bound variable, compare by level identity
- `VFlex { id; spine }` — unsolved metavariable, dispatch to `solve`
- `VNeutral { ty; neutral }` — primitive or head buried in `FIf`/`FProj` frames, decompose by head+frames

```ocaml
and value =
  | VLam of { body : closure }
  | VPi of { domain : value; codomain : closure }
  | VU
  | VAtom of Syntax.Ast.Atom.t
  | VAtomTy of atom_ty
  | VProd of value list
  | VProdTy of value list
  | VFix of { body : closure }
  | VNeutral of { ty : value; neutral : neutral }
  | VFlex of { id : meta_id; spine : spine }
  | VRigid of { lvl : lvl; spine : spine }
  | VStruct of { fields : (string * value) list }
```

### NbE engine (nbe.ml)

```ocaml
val eval           : MetaContext.t -> env -> term -> value
val apply          : MetaContext.t -> value -> value -> value
val force          : MetaContext.t -> value -> value
val quote          : MetaContext.t -> lvl -> value -> term
val conv           : MetaContext.t -> lvl -> value -> value -> bool
val closure_apply  : MetaContext.t -> closure -> value -> value
val lvl_to_ix      : lvl -> lvl -> ix
```

- `conv` is extensional: `VLam`/`VPi`/`VFix` equality introduces a fresh rigid variable and compares the bodies under it.
- Eta-equality for lambdas: `VLam` vs non-lambda η-expands the non-lambda side.

### Pattern unification (unify.ml)

`Renaming` module: maps original variable levels → solution lambda levels. For spine `[#0, #2]` (args `a` and `c` in `λa. λb. λc. ?M a c`), produces `[(0, 0); (2, 1)]` — level 0 → outermost lambda, level 2 → inner lambda.

```ocaml
val solve            : MetaContext.t -> meta_id -> spine -> value -> unit
val unify            : MetaContext.t -> lvl -> value -> value -> unit
val unify_spine      : MetaContext.t -> lvl -> spine -> spine -> unit
val unify_neutral    : MetaContext.t -> lvl -> neutral -> neutral -> unit
val unify_frames     : MetaContext.t -> lvl -> frame list -> frame list -> unit
```

Structural error type (`unify_error`): `NonLinearSpine`, `NonVariableInSpine`, `VarNotInSpine of lvl`, `NeutralVarNotInSpine of lvl`, `OccursCheck`, `CannotUnify`, `TupleLengthMismatch`, `SpineLengthMismatch`, `NeutralHeadMismatch`, `FrameMismatch`.

### Bidirectional elaboration (elaborate.ml)

`Ctx` module: groups context operations (bind, define, lookup, fresh_meta, eval, quote, unify). Name table is a `NameMap.t` (balanced tree) storing de Bruijn levels, with `lvl_to_ix` conversion at lookup time — no O(n) bumping on each binder.

```ocaml
val infer            : Ctx.t -> Surface.t -> term * value
val check            : Ctx.t -> Surface.t -> value -> term
val infer_ap         : Ctx.t -> Surface.t -> Surface.t -> term * value
val infer_let        : Ctx.t -> string -> ... -> term * value
val infer_if         : Ctx.t -> Surface.t -> Surface.t -> Surface.t -> term * value
val infer_lam        : Ctx.t -> Surface.param -> Surface.t -> term * value
val on_expr          : Surface.t -> term * value
```

Application inference: if the function type is known `VPi`, check the argument against the domain. If it's stuck (`VFlex | VRigid | VNeutral`), create fresh domain/codomain metas and unify the function type with `Pi(?dom, ?cod)`.

Structural error type (`elab_error`): `UnboundVariable of string`, `ApplyingNonFunction`, `TupleLengthMismatch`.

## Phase 2: Structs — namespace body (done)

Implemented:

- `Open` constructor in `term` (`Open of term * term`) and `Surface.t` (`Open of string * t`)

- 5 new tests: empty struct, open struct, two-field struct, nested open, field shadowing

### Tuple projection (Proj)

`FProj of int` was defined in the frame type from Phase 1 but had `failwith` stubs
in `quote_frames` and `rename/go_frames` — no term constructor created FProj frames.
Added `Proj of term * int` to complete the tuple story:

- `eval`: `Proj(e, i)` on VProd returns the i-th element; on stuck terms wraps in FProj(i)
- `quote_frames`: `FProj i` → `Proj(head, i)`
- `rename/go_frames`: `FProj i` → `Proj(head, i)`
- `elaborate`: `Proj(e, i)` infers e, forces VProdTy, returns i-th element type
- Parser: `e.0`, `e.1` postfix syntax (binds tighter than application)
- 6 new tests: proj first/second/triple, chained proj, proj from let, proj type error

## Phase 3: Struct fields + width subtyping (done)

### Three kinds of struct members

| Kind | Syntax | In type? | At construction? | Semantic |
|---|---|---|---|---|
| Constructor field | `x: I64;` | yes | required | Must be provided when constructing a record value |
| Computed field | `pub let x = e` | yes | forbidden | Derived from other fields, accessible via `.` |
| Private binding | `let x = e` | no | — | Invisible outside, only visible to later definitions inside the struct |

The struct type carries only what's visible: constructor fields + computed pub-let
fields. Private let bindings participate in neither unification nor construction.

```
struct
  x: I64;                    (* constructor field — must be provided *)
  y: I64;                    (* constructor field *)
  pub let norm = x * x + y * y   (* computed field — in type, forbidden at construction *)
  let helper = x * 2             (* private — not in type, invisible outside *)
end
```

Constructor fields must come before pub-let bindings, which must come before
private bindings. All three sections are optional.

### Width subtyping via `...` syntax

Exact struct types require all fields (both constructor and computed) to match:
```
struct x: I64; y: I64; pub let norm = I64 end
```

Partial struct types use `...` to mean "at least these":
```
struct x: I64; ... end       (* has x, possibly more constructor+computed fields *)
```

The `...` covers all remaining visible fields — constructor and computed alike.

### Unification rules

- **Exact vs exact**: strict equality of visible fields (same names, same kinds, element-wise unify)
- **Partial vs concrete**: the concrete struct must contain every field in the partial struct (kind must match), with unifiable types. Concrete can have extra fields.

### FieldAccess with stuck types

When `FieldAccess(e, name)` sees a stuck scrutinee type (VFlex/VRigid/VNeutral):
1. Create a fresh `result_ty` meta
2. Emit constraint: scrutinee type unifies with `VStruct [(name, result_ty)]` marked partial
3. Width subtyping resolves when the struct type becomes concrete

### Done

- `struct_field_kind = Field | Public | Private` — three kinds of struct members
- `Struct` term: `{ con_fields; bindings; partial }` — partial flag preserved through rename/eval
- `VStruct`: `{ fields : (string * struct_field_kind * value) list; partial : bool }`
- **Visibility**: `Dot` → non-Private (Field+Public), `Open` → Public only
- **Width subtyping**: partial VStruct unifies with any struct containing at least those
  fields. Field↔Public interchangeable in partial matching.
- **Stuck FieldAccess**: emits partial VStruct constraint with raw meta (no Bound capture),
  resolved when the struct type becomes concrete
- **raw meta**: `Ctx.raw_meta` creates `VFlex` with empty spine — avoids `VarNotInSpine`;
  `fresh_meta_val` removed. Bound capture re-introduced in Phase 7 for generalization
- **Printexc printers**: registered for `ElabError` and `UnifyError` in test executables
- Parser: `x: I64;` field declarations, `pub let` / `let` bindings, `;` separators
- 24 struct tests + 5 functor tests simulating OCaml-style module functors
  (identity, passthrough, private helper, compose, higher-order)
- 84 tests total (30 core + 54 elaborate)

### Struct bindings — `type` and `pub type`

Structs now accept `type` and `pub type` bindings alongside `let`. A `type`
binding declares a nominal ADT, scoped to the struct. `pub type` exports the
type and its constructors from the struct.

```
struct
  x: I64;
  type Color = Red | Green | Blue;   (* private ADT, visible to later bindings *)
  pub let default_color = Red;       (* uses the constructor *)
  pub type Shape = Circle | Square;  (* exported ADT *)
end
```

## Phase 4a: Nullary ADTs via `type` declarator (done)

ADTs are nominal, structs are structural — they don't mix in the same body
form.

`type` is a binder like `let`: `type Color = Red | Green | Blue in body`.
Both the type name and all constructors are in scope for the body.

### Syntax

```
type Color = Red | Green | Blue in expr
```

Inside structs, `type` and `pub type` bindings are supported (no body —
the struct scope is the body):

```
struct
  pub type Color = Red | Green | Blue;
  pub let default = Red
end
```

### Core representation

```ocaml
(* core.ml *)
type nominal_id = int  (* unique identity per type definition *)
module NominalId : sig val fresh : unit -> nominal_id end

type struct_binding_term =
  | LetBind of string * struct_field_kind * term
  | TypeBind of string * struct_field_kind * value * (string * value) list
      (* name, kind, nominal_value, [(ctor_name, ctor_value)].
         Evaluator stores values directly — no eval routing needed. *)

type term +=
  | Con of string  (* quote-only artifact, like Meta. eval scans env by name. *)

type value +=
  | VNominal of { id : nominal_id; name : string; constructors : string list }
      (* Nominal ADT type. Equality/unification compare by [id], not name.
         Two separately-defined types with the same name are distinct. *)
  | VCon of { name : string; spine : value list; nominal : value }
      (* Fully saturated constructor value. [spine = []] for nullary.
         [nominal] is the fully-applied nominal type. *)
```

`Con of string` is a quote-only artifact (like `Meta`). `eval` scans the
environment by name for a matching `VCon` or `VNominal`. Never produced by
the elaborator — only by `quote` and `rename`.

`TypeBind` carries pre-computed `VNominal`/`VCon` values. The struct
evaluator stores them directly instead of routing through `eval`. This
avoids `Con` in struct bindings — only `quote`/`rename` produce `Con`.

`VCon` uses a record shape (`{ name; spine; nominal }`) mirroring `VNeutral`.
Nullary `Red`: `VCon { name = "Red"; spine = []; nominal = color_nominal }`.

### surface.ml

```ocaml
type struct_binding =
  | LetBinding of { name : string; value : t; public : bool }
  | TypeBinding of { name : string; ctors : string list; public : bool }

type t +=
  | TypeDef of string * string list * t  (* name + ctors + body *)
```

### nbe.ml

- `eval Con name` → linear-scan env for matching `VCon`/`VNominal` by name
- `eval Struct`: `TypeBind` stores pre-computed values directly; `LetBind`
  routes through `eval` as before
- `quote VNominal n` → `Con n.name`
- `quote VCon { name; spine; _ }` → `quote_spine depth (Con name) spine`
- `quote VStruct`: fields → `LetBind` terms (type bindings are reconstructed
  as `LetBind` since the VStruct value doesn't distinguish binding kinds)
- `conv VNominal n1 VNominal n2` → `n1.id = n2.id`
- `conv VCon c1 VCon c2` → `c1.name = c2.name` + spine-length + element-wise conv

### unify.ml

- `VNominal` unifies by `id` equality only. Mismatch raises `NominalMismatch`.
- `VCon` appears in value positions, not type positions — unification only
  sees `VNominal` for types.
- `rename` for `VCon`/`VNominal`: produces `Con name` terms.

### elaborate.ml

- `TypeDef(name, ctors, body)`: creates `VNominal` with fresh `NominalId`,
  binds constructors as `VCon` + type name as `VU` in a local `body_ctx`,
  infers body in that extended context. Constructors are scoped to the body.
- `Struct` case: `TypeBinding` creates fresh `VNominal`, binds constructors
  and type name in the struct's internal scope. Produces `TypeBind` with
  pre-computed `VCon` values for the evaluator. Struct type fields: type
  name → `VU`, constructors → `VNominal`.

### Parser

- `type ID = ID {| ID} in expr` — top-level type definition with body.
- `struct {pub,} type ID = ID {| ID}` — type binding inside struct.
- No `[]` brackets — type params are space-separated (Phase 4b).

### Implementation notes

- **Nominal identity**: `NominalId.fresh()` global counter. Each `type`
  definition gets a unique `nominal_id`. Two separately-defined types with
  the same name won't unify.
- **`Con of string`**: quote-only artifact (like `Meta`). `eval` scans the
  env by name — works because quoted terms are evaluated in closures that
  capture the elaboration environment where constructor/type bindings are
  present. Elaborator never produces `Con` — only `Var ix` for constructor
  refs and `TypeBind` for struct type bindings.
- **`struct_binding_term`**: separates let-binding eval (routed through
  `eval`) from type-binding eval (pre-computed values stored directly). This
  keeps `Con` out of struct bindings entirely.
- **Context threading**: `infer`/`check` return extended contexts. This was
  needed when `TypeDef` had no body (constructors leaked to caller), but is
  kept because it's mechanically simple and prevents a class of bugs where
  context-extending constructs would see stale environments.

### Tests (10 new, 65 total elaborate)

- Constructor type, multiple constructors, shadowing
- Undefined constructor error
- Type name accessible in body (`Color : Type`)
- `pub type` inside struct, constructor via dot
- Private type not visible via dot, visible to later binding
- Two structs: distinct nominal IDs, cross-struct annotation rejected

## Phase 4b: Parameterized ADTs with payloads (next)

Constructors gain type parameters and payload arguments. In the DT setting,
constructors become Pi-typed functions — no special `Con` term, just `VPi`-
typed lambdas that bottom out at `VCon` (value level only, for pattern matching).

```
type Option = fun a -> ...
Option : (a : Type) -> Type           (* a Pi type *)
Some    : (a : Type) -> a -> Option a
None    : (a : Type) -> Option a

Ap(Ap(Var "Some", I64), 42)          (* creates the nominal value *)
```

`Some` evaluates to `VLam(a, VLam(x, VCon("Some", [a; x], option_ty a)))`.
Applying it to `I64` gives `VLam(x, VCon("Some", [I64; x], option_ty I64))`.
Applying that to `42` gives `VCon("Some", [I64; 42], option_ty I64)` — a
fully saturated constructor value that `VCon` carries the raw materials for
pattern matching (Phase 5) to dispatch on.

#### Syntax

```
type Option a = Some a | None in body
type Tree a = Leaf a | Node (Tree a) (Tree a) in body
type Result a e = Ok a | Err e in body
```

Type params are regular function parameters — `Option : (a : Type) -> Type`,
applied as `Option I64`. Constructors are functions too: `Some I64 42 : Option I64`,
`None I64 : Option I64`. In this phase all type arguments are explicit — the user
writes them. Implicit arguments are deferred to Phase 6.

#### Core representation

```ocaml
(* core.ml — VNominal gains params; VCon spine collects type+value args *)
type value +=
  | VNominal of {
      id : nominal_id;
      name : string;
      params : value list;                 (* type parameters, e.g. [VAtomTy I64] *)
      constructors : (string * value option) list;
          (* constructor name + optional payload type. None = nullary. *)
    }
  (* VCon unchanged from 4a: { name; spine; nominal } *)
  (* VCon { name = "Some"; spine = [VAtomTy I64; VAtom 42]; nominal = ... } *)
```

Pi and Ap do NOT get explicitness flags yet — that arrives in Phase 6.
All arguments are explicit for now.

#### Implementation steps

1. **`surface.ml`** — extend `TypeDef` with type param names and constructor
   payload type annotations:
   `TypeDef of string * string list * (string * t option) list * t`
   (name + param names + (ctor_name * payload_type option) list + body).

2. **`core.ml`** — extend `VNominal` with `params : value list` and typed
   constructor entries (`(string * value option) list` — `None` = nullary).
   No changes to `Pi`/`Ap` — all arguments are explicit for now.

3. **`nbe.ml`** — each constructor `C` with payload type `P` gets a curried
   VLam chain: type params first (each `(a : Type) -> ...`), then payload
   (`(p : P) -> ...`), body is `VCon{name=C; spine; nominal}`.
   Eval/apply via normal application — no special case.
   `conv VCon`: name + spine-length + element-wise conv.

4. **`unify.ml`** — `VNominal` unifies by `id` (from 4a); param lists unified
   element-wise. `VCon` not unified directly (values, not types).

5. **`elaborate.ml`** — `TypeDef(name, params, ctors, body)`:
   - For each param `'a`, create a fresh meta `?a : Type`.
   - Build `VNominal{id; name; params=[?a; ...]; constructors=...}`.
   - For each constructor, build the curried Pi type (all `Explicit`).
   - Store constructor as `VLam` chain in context; bind type name as `VU`.
   - Infer body in extended context.
   - Constructor application via normal `infer_ap`.

6. **Parser** — `type ID a b = ConA a | ConB a b | ... in expr`.
   Constructor payloads: `Con expr` (unary) or `Con(expr1, expr2)` (tuple).

#### Recursive types

`type Tree a = Leaf a | Node (Tree a) (Tree a) in body` — the type name
`Tree` appears in its own constructor payloads. The elaborator handles this
via the `Fix` term already in core: the type definition creates a `Fix`-
wrapped nominal that can reference itself.

#### Test plan

- Parameterized ADT with unary payload, construct via `Some I64 42`.
- ADT with multiple type params (`Result a e`).
- Recursive ADT (`Tree a`) — `Node I64 (Leaf I64 1)`.
- Constructor partial application: `Some I64` has type `I64 -> Option I64`.
- Two distinct ADTs with same-named constructors — correct nominal type via
  context lookup.

## Phase 5: Pattern matching

Pattern matching on nominal ADTs, with exhaustiveness checking and nested
patterns. This is a separate phase from 4a/4b because it touches nearly
every part of the pipeline and brings substantial complexity.

### Surface

```ocaml
(* surface.ml *)
type t +=
  | Match of t * (pat * t) list            (* match scrutinee with branches *)

and pat =
  | SPatCon of string * spat list           (* constructor with sub-patterns *)
  | SPatBind of string                      (* variable binding *)
```

### Core

```ocaml
(* core.ml *)
type term +=
  | Match of term * (pat * term) list       (* pattern matching *)

and pat =
  | PatCon of string * pat list             (* constructor with sub-patterns *)
  | PatBind of string                       (* variable binding *)
```

`Match` evaluates the scrutinee to a `VCon`, then dispatches on the
constructor name. Sub-patterns bind the spine elements from `VCon`.

### Implementation steps

1. **`surface.ml`** — add `Match`, `SPatCon`, `SPatBind`.

2. **`core.ml`** — add `Match`, `PatCon`, `PatBind` to `term`.

3. **`nbe.ml`** — `eval Match(scrutinee, branches)`:
   - Eval scrutinee to `VCon(cname, spine, _)`.
   - Linear-scan branches for `PatCon(cname, subpats)`.
   - Extend environment with spine elements bound to sub-patterns.
   - Eval the body.
   - `PatBind` matches anything and binds the scrutinee value.
   - `quote` reconstructs `Match` nodes from decision trees (if compiled).

4. **`unify.ml`** — no changes. `VCon` is a value, not a type — unification
   only ever sees `VNominal`.

5. **`elaborate.ml`** — `infer(Match(scrutinee, branches))`:
   - Infer scrutinee type → force to `VNominal`.
   - For each `(SPatCon(c, subpats), body)`:
     - Look up `c`'s payload type in the nominal's constructor list.
     - Bind sub-pattern variables to their corresponding payload types.
     - Infer body.
   - Unify all branch result types.
   - `check` variant: check each branch against the expected type.

6. **Parser** — `match expr | pat -> expr {| pat -> expr} end` syntax.
   Pattern disambiguation: `ID` → `SPatCon(ID, [])` if the name is a known
   constructor, `SPatBind(ID)` otherwise.

### Exhaustiveness checking

Reuse the Maranget-style missing-pattern machinery from
`lib/semantic/match/` (matrix, missing_pat, match_compile), adapted to
operate on `PatCon` patterns instead of the old `Con` patterns.

The decision tree compilation from the same library (decision_tree.ml,
matrix decomposition by constructor/literal) ports directly — `PatCon`
replaces the old constructor pattern, `PatBind` is the wildcard.

### Test plan

- Nullary match: `match Red | Red -> 1 | Green -> 2 | Blue -> 3 end`.
- Catch-all variable pattern.
- Parameterized constructor match with nested sub-patterns.
- Exhaustiveness warning for missing constructors.
- Nested patterns: `match expr | Some(Some(x)) -> x | None -> default end`.
- Two ADTs with overlapping constructor names — dispatch via nominal type.

## Phase 6: Implicit arguments

Make type arguments implicit for constructors like `None` and `Some`
via `{a : Type}` Pi syntax. The elaborator inserts metas or propagates
expected types to fill implicit arguments automatically.

### Core representation

```ocaml
(* core.ml — Pi and Ap get an explicitness flag *)
type explicitness = Implicit | Explicit

type term +=
  | Pi of explicitness * term * term    (* (x : A) -> B  or  {x : A} -> B *)
  | Ap of explicitness * term * term    (* f arg  or  f {arg} *)

type value +=
  | VPi of { explicitness : explicitness; domain : value; codomain : closure }
```

### Syntax

```
None : {a : Type} -> Option a       (* a is implicit *)
Some : {a : Type} -> a -> Option a  (* a implicit, payload explicit *)

None           (* elaborator inserts meta for {a} *)
None {I64}     (* override: explicit implicit application *)
```

### Implementation steps

1. **`core.ml`** — add `explicitness` to `Pi` and `Ap`; `VPi` gains explicitness.

2. **`surface.ml`** — `App of t * t * explicitness` for `f {arg}` syntax.

3. **`nbe.ml`** — `eval`/`apply`/`quote` handle explicitness on Ap/VPi.

4. **`unify.ml`** — `rename` handles `Pi(expl, ...)`/`Ap(expl, ...)`.

5. **`elaborate.ml`** — three cases for implicit arg resolution:
   - **Explicit `f {arg}`**: elaborator forces next Pi to be `Implicit`, checks
     `arg` against domain.
   - **Expected type available** (`check None (Option I64)`): elaborator
     propagates `Option I64` to solve `a = I64`, no meta left.
   - **No expected type** (`infer None`): fresh meta `?m : Type`,
     `Ap(Implicit, None, InsertedMeta(?m))`, type `Option ?m`. Meta stays
     unsolved for Phase 7 generalization.

6. **Parser** — `f {arg}` for explicit implicit application.

### Test plan

- `None {I64}` explicit override.
- `Some 42` with expected type `Option I64` → infers `a = I64`.
