# Generated symbol cleanup plan

## Context

Elaboration currently encodes several semantic concepts as generated string names, private marker fields, or string-encoded atoms. This works for small examples, but it is brittle: user names can collide with generated names, module `open` has to rediscover metadata by parsing values, and ordinary struct/module fields are overloaded to carry compiler-only state.

The goal is to replace these conventions with explicit structural representations in the core/value model and elaboration context.

## Current generated symbols

### Trait declaration markers

Current shape in `lib/semantic/typecheck/elaborate.ml`:

```ocaml
"trait:" ^ string_of_int trait_info.trait_id ^ ":" ^ trait_info.trait_name
```

Trait declarations elaborate to `VAtom (Atom.String marker)`, and opening a module attempts to parse this marker back into `trait_info` through a global registry.

Problems:

- Trait metadata is represented as an ordinary string value.
- The marker is parsed with string splitting.
- The global registry has to stay synchronized with values that may be stored in modules.
- Module opening has to know about this encoding.

### Trait dictionary marker fields

Current trait dictionary types are ordinary `VStruct` values with private fields:

```ocaml
("__arg", Private, arg)
("__trait", Private, VAtom (Atom.String trait_name))
("__trait_id", Private, VAtom (Atom.I64 trait_id))
```

`resolve_trait_dict_ty` recognizes trait dictionaries by matching this exact field prefix.

Problems:

- Trait dictionary types are not structurally distinct from user structs.
- The field order and field names are semantic invariants.
- This makes the struct representation carry compiler metadata that users should not be able to model accidentally.

### Generated impl binding names

Current impl elaboration creates names like:

```ocaml
"__impl_" ^ trait_name
```

The generated binding stores the impl dictionary value so trait evidence can point at it by de Bruijn level.

Problems:

- Generated names can collide with user names unless they are globally reserved.
- Impl dictionaries are not user-facing values but are represented as ordinary bindings/module fields.
- Public/private impl visibility is mixed with ordinary field visibility.

### Effect operation grouping keys

Effect handler coverage currently builds keys like:

```ocaml
string_of_int eff.id ^ ":" ^ op_name
```

Problems:

- The key is naturally a pair `(effect_id, op_name)`, not a string.
- This is local and low risk, but it is unnecessary string encoding.

### Anonymous `_` binders

Some elaboration paths use:

```ocaml
Ctx.bind ctx "_" ty
```

for synthetic binders when constructing dependent function codomains or metas.

Problems:

- `_` is a surface wildcard spelling, but these are compiler-created binders.
- `Ctx.bind` adds the name to the name table, so this can affect lookup/shadowing behavior.
- There is already `Ctx.bind_anonymous`, but it is not used everywhere this pattern appears.

### Recursive `Self` placeholder through `HPrim`

Parameterized record `Self` handling currently creates neutrals with:

```ocaml
HPrim name
```

where `name` is the record name.

Problems:

- `HPrim` also represents real primitives such as `+`, `panic`, and `eq_i64`.
- A recursive record type placeholder is not a primitive.
- Conversion/unification treats `HPrim` by string equality, so this relies on record names not colliding with primitive names or other symbolic heads.

### Dead marker helper

`trait_marker_ty` appears unused and still constructs older private marker fields. It should be deleted once the trait representation is replaced.

## Target representation

### 1. Separate trait descriptors from ordinary values

Add an explicit semantic representation for trait declarations. Possible shape:

```ocaml
type trait_decl = {
  trait_id : int;
  trait_name : string;
  trait_params : string list;
  trait_fields : (string * closure) list;
}

and value =
  | ...
  | VTrait of trait_decl
```

Then trait declarations can elaborate to `VTrait trait_info` instead of `VAtom (Atom.String "trait:...")`.

Open/import logic can register traits by pattern matching on `VTrait`, not by parsing marker strings.

### 2. Represent trait dictionary types directly

Add a dedicated trait dictionary type representation instead of encoding dictionaries as `VStruct` with magic private fields.

Possible shape:

```ocaml
and value =
  | ...
  | VTraitDictTy of {
      trait : trait_decl;
      args : value list;
      fields : (string * value) list;
    }
```

The dictionary value itself can remain a struct-like value containing public methods, but the dictionary type should be structurally distinct.

This makes `resolve_trait_dict_ty` unnecessary or reduces it to a direct pattern match.

### 3. Store impl evidence in a layered map

Do not create `__impl_<Trait>` names. Trait implementations should be tracked in a separate dictionary-like evidence structure, not as ordinary module fields.

Conceptually, the elaboration context should contain a layered map:

```ocaml
impls[trait_id][trait_arg_key] = implementation_dictionary
```

or, preserving the current level-based evidence representation:

```ocaml
trait_id -> arg_key -> trait_evidence list
```

where each `trait_evidence` points to the implementation dictionary by de Bruijn level or, for imported/opened module evidence, by a structural module evidence entry.

For local expression-level impls, add anonymous definitions:

```ocaml
Ctx.define_anonymous : Ctx.t -> value -> value -> Ctx.t * name_entry
```

`elaborate_impl` can define the dictionary value anonymously and store the returned level in the impl map.

For module-level impls, module values/types should carry a separate evidence section instead of ordinary public fields:

```ocaml
type module_entry =
  | ModuleValue of string * struct_field_kind * value
  | ModuleTrait of string * struct_field_kind * trait_decl
  | ModuleImpl of struct_field_kind * trait_evidence
```

Opening a module merges its public impl evidence into the current context's impl map. Private impls are available only while elaborating later bindings in the defining module.

### 4. Make effect operation keys structural

Replace string keys with pair keys:

```ocaml
module EffectOpKey = struct
  type t = int * string
  let equal (a_id, a_op) (b_id, b_op) = a_id = b_id && String.equal a_op b_op
  let hash = Hashtbl.hash
end

module EffectOpTable = Hashtbl.Make (EffectOpKey)
```

Then use `(eff.id, branch.op)` directly.

### 5. Stop binding synthetic `_` names

Replace internal `Ctx.bind ctx "_" ty` uses with an anonymous context extension helper.

If a closure needs an extended environment but no user-visible name, prefer:

```ocaml
let ctx', _entry = Ctx.bind_anonymous ctx ty
```

or introduce a clearer helper:

```ocaml
Ctx.bind_internal : Ctx.t -> value -> Ctx.t
```

This should not add anything to `name_table`.

### 6. Represent recursive `Self` directly

Recursive record elaboration should not encode `Self` as a fake primitive neutral head:

```ocaml
VNeutral { neutral = { head = HPrim name; frames = ... }; _ }
```

Use a dedicated semantic value and internal core term instead:

```ocaml
| VSelfType of value list
| SelfTypeRef of term list
```

The value means “the record type currently being elaborated, applied to the current type parameters.” It has no name identity and is not a primitive.

## Implementation order

### Phase 1: low-risk local cleanup

- Replace effect handler grouping strings with structural `(effect_id, op_name)` keys.
- Delete unused `trait_marker_ty` if no references remain.
- Replace synthetic `Ctx.bind ctx "_" ...` sites with anonymous/internal context extension where safe.

These changes should not alter language behavior.

### Phase 2: remove generated impl names

- Replace the flat `trait_evidence` list with a layered impl map keyed first by trait id and then by trait argument key.
- Add `Ctx.define_anonymous` for defined values that need de Bruijn levels but no source name.
- Change expression-level `elaborate_impl` to define impl dictionaries anonymously and insert evidence into the impl map.
- Add structural module impl entries so module-level impls no longer become ordinary `__impl_<Trait>` fields.
- Make `open` merge public module impl evidence into the current context's impl map.

Expected result: no more `__impl_` bindings, and impl lookup behaves like `impls[Eq][I64]` rather than field lookup by generated name.

### Phase 3: structural trait dictionaries

- Add `VTraitDictTy` to `Core.value`.
- Update debug printing, quote, substitution/traversal, NbE conversion, and unification for the new variant.
- Change `trait_dict_ty` to return `VTraitDictTy`.
- Replace `resolve_trait_dict_ty` with direct pattern matching.
- Remove `__arg`, `__trait`, and `__trait_id` from dictionary types.

Expected result: trait dictionary types can no longer be confused with user structs.

### Phase 4: structural trait declarations

- Add `VTrait` or equivalent trait descriptor value.
- Change `elaborate_trait` to return a structural trait value instead of a string atom marker.
- Update module elaboration and `open_module_value` to register public traits by matching `VTrait`.
- Remove `trait_marker`, `trait_info_of_marker`, and the global marker parsing path.

Expected result: no more `"trait:<id>:<name>"` markers.

### Phase 5: replace `HPrim` placeholders

- Add `VSelfType` and `SelfTypeRef` for recursive record `Self` during type elaboration.
- Update quote, evaluation, conversion, unification, substitution, and traversal for the new form.
- Keep actual primitives separate from record/type placeholders.

Expected result: recursive record `Self` no longer reuses primitive-head identity and does not carry a name-based placeholder.

## Tests to add or update

- A user binding named `__impl_Eq` should not interfere with trait impl resolution.
- A user record/struct with fields named `__arg`, `__trait`, and `__trait_id` should not be recognized as a trait dictionary.
- Opening a module with a public trait should still make that trait available.
- Opening a module with a public impl should still make evidence available.
- Private traits/impls inside modules should not leak through open/import.
- Existing equality stdlib behavior should remain:
  - builtin atom equality works through `Eq` impls;
  - nominal/record equality without impl fails during elaboration.
- Recursive `Self` examples should still elaborate and evaluate after replacing the `HPrim` placeholder.
