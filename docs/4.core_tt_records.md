# core_tt record construction and record patterns

## Context

Record types are existing `struct` types. The distinction is between a struct/module/type value and an instance of that struct type, which is a record value.

This plan does **not** add a separate record type declaration syntax. Instead, it adds record instance construction and pattern matching for existing struct types.

Target syntax:

```fun
let Point = struct x: I64; y: I64; end in
match Point {x = 3; y = 4} with
  Point {x; y} -> x + y
end
```

Partial, renamed, and refutable field patterns should also work:

```fun
Point {x; _}
Point {x = n; y}
Point {flag = true; value}
```

## Recommended approach

Reuse `VStruct` as the type of record instances, and introduce a separate runtime value for record instances. `struct ... end` continues to produce a struct/module/type value. `Point {x = 1; y = 2}` constructs a record instance whose type is the `VStruct` value bound to `Point`.

This keeps the language model aligned with the main pipeline: record types are structs, record values are field-value records, and record patterns destructure record instances by named field occurrence.

## Implementation steps

1. Add record construction and pattern surface forms.
   - In `lib/semantic/core/surface.ml`, add:
     - `RecordConstruct of { typ : t; fields : (string * t) list }` for expressions like `Point {x = 1}`.
     - `PatRecord of { typ : string; fields : (string * pat option) list; partial : bool }` for patterns like `Point {x; _}`.
   - Keep `Struct` unchanged; it remains the type/module form.

2. Extend the core test parser.
   - In `lib/semantic/core/test/core_parser.mly`, parse record construction in expression application position:
     - `callee {x = expr; y = expr}` becomes `RecordConstruct { typ = callee; fields }`.
     - Preserve existing implicit application `f {arg}`; disambiguate record construction by looking for `ID EQUALS` entries inside braces.
   - Parse record patterns in `pat_atom`:
     - `Point {x; y}`,
     - `Point {x = pat; y}`,
     - `Point {x; _}`.
   - Validate `_` appears only as the final record-pattern entry and that at least one named field is present.

3. Add core term/value/pattern support.
   - In `lib/semantic/core/core.ml`, add:
     - a term form for record construction, e.g. `RecordConstruct of { typ : term; fields : (string * term) list }`,
     - a value form for record instances, e.g. `VRecord of { typ : value; fields : (string * value) list }`,
     - a pattern form `CPatRecord of { fields : (string * core_pat) list; partial : bool }`.
   - `VStruct` remains the type/module representation; `VRecord` is the instance.

4. Elaborate record construction.
   - In `lib/semantic/core/elaborate.ml`, infer/check the `typ` expression and require it to evaluate to `VStruct` with visible constructor fields (`Field`).
   - Check supplied fields against the struct field types.
   - Reject missing fields, duplicate fields, and unknown fields for construction.
   - Return a core `RecordConstruct` term with type equal to the `VStruct` type value.
   - Leave public/private module fields out of required record construction unless they are `Field` entries.

5. Elaborate record patterns.
   - In `lib/semantic/core/elaborate.ml`, extend `elaborate_pat_binders` for `PatRecord`.
   - Resolve the pattern type name via `Ctx.lookup`, require it to be a `VStruct`, and unify the scrutinee type with that `VStruct`.
   - For each listed field:
     - look up its `Field` type in the `VStruct`,
     - shorthand `Point {x}` elaborates as a field `CPatBind` and returns binder `(x, field_ty)`,
     - explicit `Point {x = pat}` elaborates `pat` against the field type.
   - If `partial = false`, require all `Field` entries from the struct type to be listed; allow missing fields only with trailing `_`.
   - Reject unknown fields and duplicate fields.
   - Extend `refine_match_scrutinee_ty` so unknown scrutinee types can be refined from `PatRecord` by resolving its struct type name.

6. Add named field occurrences.
   - In `lib/semantic/core/core_decision_tree.ml`, add `OField of { parent : occurrence; name : string }` and update occurrence equality/hash.
   - In `lib/semantic/core/elaborate.ml`, update `type_at_occurrence` for `OField` by looking through `VStruct` field types.

7. Lower record patterns in the core match compiler.
   - In `lib/semantic/core/core_match_compile.ml`, add a record-pattern specialization analogous to tuple/product specialization:
     - collect field names from active `CPatRecord` patterns,
     - create `OField { parent = occ0; name }` columns,
     - replace each `CPatRecord` row with its field subpatterns, using `CPatWild` for omitted fields in partial patterns,
     - expand `CPatWild` rows to field wildcards,
     - expand `CPatBind` rows to field wildcards and bind the whole record occurrence.
   - No new decision-tree node should be needed; nested field subpatterns reuse existing `Switch`, `Destruct`, `ProductPat`, and `OrPat` logic.

8. Update NbE runtime support.
   - In `lib/semantic/core/nbe.ml`, evaluate `RecordConstruct` into `VRecord`.
   - Add `Dot` support for `VRecord` by looking up instance fields; keep `VStruct` field access behavior unchanged for modules/types.
   - Make `eval_match` handle `VRecord` with `compile_with_domains`, like `VProd`.
   - Extend `resolve_occurrence_opt` for `OField` by looking up fields in `VRecord`.
   - Extend runtime `domain_of_occurrence` to return domains for record field values.
   - Extend `quote`, `conv`, and `conv_pat` for record construction/values/patterns.

9. Add focused tests.
   - In `lib/semantic/core/test/test_elaborate.ml`, add tests for:
     - record type via `let Point = struct x: I64; y: I64; end`,
     - record construction `Point {x = 1; y = 2}`,
     - record field access on an instance,
     - shorthand pattern `Point {x; y}`,
     - field-order independence `Point {y; x}`,
     - rename/subpattern `Point {x = n; y}`,
     - partial `Point {x; _}`,
     - incomplete complete pattern rejection,
     - unknown/duplicate field rejection,
     - literal field dispatch and finite-field exhaustiveness.
   - In `lib/semantic/core/test/test_core.ml`, add runtime tests for construction, field access, shorthand binding, reordered fields, and literal field dispatch.

10. Update checklist after validation.
   - After targeted and broad validation pass, mark `Record patterns — Foo {x; y} or Foo {x; _} destructuring` done in `docs/3.wiring_checklist.md`.

## Scope notes

- Do not add a separate record type declaration syntax; record types are existing `struct` type values.
- Do not make module/namespace structs pattern-matchable as records unless they are used as the type in a record construction/pattern.
- Parameterized struct record types can be deferred if they require substantial extra work; start with non-parameterized `let Point = struct ... end` record types.
