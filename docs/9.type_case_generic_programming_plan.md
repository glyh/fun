# Type-case / generic programming plan

## Goal

Add useful generic programming over runtime `Type` values by extending existing primitive type-head matching with branch-local dependent refinement. The first milestone should let a match on `T : Type` refine values whose type depends on `T`, while keeping nominal type-head matching for a later phase.

## Decisions

- Type-case is a normal runtime match over `Type` values.
- Matches over open `Type` require a fallback because the type universe is infinite.
- The first implementation phase focuses on primitive type heads: `I64`, `Bool`, `Unit`, and `Char`.
- Nominal type-head matching is planned separately after primitive refinement works.
- In a primitive type-head branch, the checker may treat the matched type variable as equal to the matched primitive type.
- The bottom/absurd type is named `Absurd` (not `Void`, to avoid cross-language confusion with C-style `void` ≈ `Unit`).

## Phase 1: Primitive dependent refinement ✓

- [x] branch-local refinement: `T` narrows to the matched primitive head inside each branch
- [x] dependent variables typed by `T` are usable as the refined type (e.g. `x : T` becomes `x : I64`)
- [x] `refine_branch_context` replaces the matched type variable in context entries
- [x] `refine_branch_expected` replaces the matched type variable in the expected result type
- [x] `refinement_target_of_scrutinee` detects `Var` scrutinees without evaluation
- [x] all three match paths wired: `infer`, `check` (VU scrutinee), `check` (ordinary scrutinee), and `collect_effects`
- [x] lambda bodies treated as pure in `collect_effects` to avoid spurious unification

```fun
let is_zeroish : {T : Type} -> T -> Bool = fun {T : Type} -> fun x ->
  match T with
  I64 -> x == 0
  | Bool -> x == false
  | Unit -> true
  | Char -> x == 'a'
  | _ -> false
  end
```

Tests:

- [x] `is_zeroish 0` evaluates to `true`, `is_zeroish 1` → `false`
- [x] `is_zeroish false` → `true`, `is_zeroish true` → `false`
- [x] `is_zeroish ()` → `true`, `is_zeroish 'a'` → `true`
- [x] branch with wrong refined operation (e.g. `x == false` in `I64` branch) is rejected

## Phase 2: Absurd type and string support

Before refined return types, add the `Absurd` bottom type and `String` type so `panic` can carry a message and uninhabited branches are expressible.

### 2a: Absurd type ✓

- [x] add `TAbsurd` to `atom_ty` in `core.ml`
- [x] add `Absurd` as a type pattern name in `bare_pat_name`
- [x] wire into `init_ctx` as a `Type`-kinded primitive
- [x] no value constructors — `Absurd` is uninhabited
- [x] `all_atoms` returns `Some []` for exhaustive checking over empty domain

### 2b: String type ✓

- [x] add `TString` to `atom_ty` in `core.ml`
- [x] add `String of string` to `Atom.t` in `atom.ml`
- [x] lexer already reads `"..."` into `STRING` tokens; wired into parser as `Atom (String s)`
- [x] add `String` as a type pattern name in `bare_pat_name`
- [x] wire `String` into `init_ctx` as a `Type`-kinded primitive
- [x] `==` accepts same-typed string operands via stdlib dispatch
- [x] no concatenation or other string operations in this pass

### 2c: Update panic ✓

- [x] change `panic` type from `{T : Type} -> Unit -> T` to `{T : Type} -> String -> T`
- [x] runtime extracts the string argument from the second frame and raises `EvalError msg`
- [x] update `init_ctx` prims and `prim_table`/neutral handler

## Phase 3: Refined return types ✓

- [x] `default {I64}` evaluates to `0`
- [x] `default {Bool}` evaluates to `false`
- [x] `default {Unit}` evaluates to `()`
- [x] `default {String}` raises `EvalError "no default"`
- [x] `default` type-checks as `{T : Type} -> T`
- [x] branch expected types refine with the matched type (`refine_branch_expected` already wired from Phase 1)

## Phase 4: Nominal type heads ✓

Extend type-case refinement to nominal type heads after primitive heads are stable.

- [x] fully-applied nominal type-head patterns match by stable nominal ID, e.g. `Option I64`
- [x] nominal parameter subpatterns can bind lowercase type variables, e.g. `Option x`
- [x] complex parameter subpatterns are supported, e.g. `Option (Option I64 | I64)`
- [x] unresolved uppercase names stay concrete and reject instead of becoming binders
- [x] decision-tree switches include nominal IDs and specialize nominal parameters as child occurrences
- [x] runtime `VNominal` values preserve declaration IDs and expose parameter values for pattern matching

Example:

```fun
type Option A = Some A | None in
let classify : Type -> I64 = fun T ->
  match T with
  | I64 -> 1
  | Option I64 -> 2
  | Option x -> match x with I64 -> 3 | _ -> 4 end
  | _ -> 0
  end
```

## Phase 5: Generic programming library surface ✓

Once primitive and nominal refinement are in place, add small library-style examples that justify the feature:

- [x] type-directed equality dispatch via existing `(==)` / `(!=)` over supported primitive heads;
- [x] type-directed defaulting with a supplied fallback value, demonstrated as user-land `default_or`;
- [x] type-directed classification to string literals under the current string surface;
- [x] nominal type-head classification over applied ADTs, including `Option I64` and `Option _`;
- [ ] protocol/typeclass bridge experiments after the protocol design exists.

## Non-goals for now

- Macro integration.
- Protocol/typeclass implementation.
- Nominal type-head matching in the first milestone.
- Compile-time-only type-case erasure.
- Exhaustive matching over all `Type` values.
- A universal default value for arbitrary types.
- String concatenation or other string operations beyond equality and literals.
