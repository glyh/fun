# Trait modules and Eq stdlib plan

## Context

The goal is to replace the current type-case polymorphic equality in stdlib with trait-based `Eq`, while keeping stdlib clean and keeping tests that exercise type-case independent via local helpers. The first attempted direction—putting `trait Eq` / `impl Eq ...` into the embedded `stdlib_source` and extending `init_ctx`'s stdlib walker—would make stdlib more special-cased. Instead, traits and impls should become normal module/struct definitions so stdlib can be sourced explicitly through `Core_loader`.

This also exposes a broader issue: module-level `pub` semantics need to apply to trait and impl declarations, not just ordinary values/effects/types.

## Current state

- Module files are parsed as `Surface.Struct` via `Core_lexer.parse_module` / `Core_parser.module_eof`.
- `Surface` already has `TraitBinding` and `ImplBinding` variants for struct/module bindings.
- The parser already accepts:
  - `pub trait ... = struct ... end`
  - private `trait ... = struct ... end`
  - `pub impl ... = struct ... end`
  - private `impl ... = struct ... end`
- Expression-level `TraitDef` / `ImplDef` are scoped `... in body` forms and currently work by modifying the elaboration context only for the body.
- Struct/module elaboration currently rejects `TraitBinding` and `ImplBinding` with `ApplyingNonFunction`.
- Imports already go through `Core_loader` and infer the parsed module struct.
- Ordinary module visibility is represented with `Public`, `Private`, `Method`, and `PrivateMethod`; field access and `open` already respect those kinds.

## Recommended implementation

### 1. Do not extend the embedded stdlib walker

Do not add `TraitDef` / `ImplDef` cases to `init_ctx`'s `add_stdlib` walker in `lib/semantic/typecheck/elaborate.ml`.

Instead, make trait/impl declarations work through normal module/struct elaboration and have tests explicitly source a stdlib module through `Core_loader`.

The accidental embedded `stdlib_source` rewrite should be reverted or treated as temporary until the explicit stdlib module path replaces it.

### 2. Elaborate trait bindings in structs/modules

In `lib/semantic/typecheck/elaborate.ml`, replace the current rejection of `Surface.TraitBinding` inside `Surface.Struct` with real handling:

- Call existing `elaborate_trait`.
- Add the resulting `trait_info` to `ctx.traits` for subsequent bindings in the same struct/module.
- Define the trait name in the local value context using the existing `trait_ty`, so subsequent declarations can refer to it.
- Respect `public`:
  - `pub trait` should be exposed in the inferred module type and available through import/open.
  - private `trait` should be usable by later declarations inside the same module but not visible outside.

### 3. Elaborate impl bindings in structs/modules

In `lib/semantic/typecheck/elaborate.ml`, add handling for `Surface.ImplBinding` inside `Surface.Struct`:

- Call existing `elaborate_impl` for local trait names, matching current `ImplDef` behavior.
- Add the resulting evidence to the context for later bindings in the same module.
- Emit the implementation dictionary as a synthetic core binding so runtime field values remain consistent.
- Respect `public`:
  - A public impl should be available to users who explicitly import/open the module according to the module visibility rules.
  - A private impl should support later declarations in the same module but should not satisfy trait bounds outside the module.

Because impl evidence is not a user-facing value, prefer a hidden/synthetic binding convention rather than exposing dictionaries as ordinary public fields.

### 4. Extend import/open to register public trait metadata and impl evidence

The existing `open` path imports ordinary public fields into the value context. Trait metadata and impl evidence also live in elaboration context fields (`ctx.traits` and `ctx.trait_evidence`), so imported/opened modules need to re-register public trait and impl declarations there.

Required behavior:

- `pub trait Eq ...` in module `M` can be used as `M.Eq` where syntax supports qualified paths, or via `open M in ...`.
- private `trait Eq ...` in module `M` is not visible to import users.
- `pub impl Eq I64 ...` can satisfy an external `{A : Eq}` bound only after the module is explicitly imported/opened.
- private impls never leak outside the module.
- Private traits/impls can still be used by later public definitions inside their defining module.

### 5. Keep or improve qualified trait paths deliberately

The parser already accepts dotted impl trait heads, but elaboration currently rejects non-empty `trait_path` for `ImplDef`.

Decide whether this phase should support qualified trait impls such as:

```fun
impl M.Eq I64 = struct ... end
```

If not, keep the existing rejection but add tests that document the current limitation. If yes, reuse `resolve_path_core_value`/module lookup behavior and add trait metadata lookup by qualified path.

### 6. Move Eq stdlib into explicit module source for tests

Define stdlib as a module source in tests, for example:

```fun
pub trait Eq A = struct
  eq : A -> A -> Bool
end

pub impl Eq I64 = struct let eq x y = eq_i64 x y end
pub impl Eq Bool = struct let eq x y = eq_bool x y end
pub impl Eq Char = struct let eq x y = eq_char x y end
pub impl Eq Unit = struct let eq x y = eq_unit x y end
pub impl Eq String = struct let eq x y = eq_string x y end

pub let (==) = fun {A : Type} lhs rhs -> Eq.eq lhs rhs
pub let (!=) = fun {A : Type} lhs rhs -> not ((==) {A} lhs rhs)
```

The exact annotations may need to stay explicit as:

```fun
pub let (==) : {A : Eq} -> A -> A -> Bool = ...
```

if module binding syntax is extended to support typed `pub let` declarations.

The primitive impls must call `eq_i64`, `eq_bool`, `eq_char`, `eq_unit`, and `eq_string`; they must not call `(==)` recursively.

### 7. Update equality and type-case tests

Tests that want stdlib equality should explicitly load/open the stdlib module through `Core_loader`.

Tests that specifically exercise type-case refinement should not depend on stdlib `==`. Where needed, add a local helper using the old type-case equality shape, e.g. a local `type_eq`/`same_typecase` function that matches on `A : Type` and dispatches to primitive equality.

Expected semantic updates:

- Primitive equality still works through `Eq`.
- Polymorphic helpers use `{A : Eq}` rather than `{A : Type}`.
- Unsupported nominal/record equality should become an elaboration error due missing `Eq` impl, replacing the old runtime panic expectation.
- Effect/type equality tests should either use a local type-case helper or be rewritten as direct conversion/elaboration assertions, depending on what they intend to test.

## Files to change

- `lib/syntax/surface.ml`
  - Likely no structural changes needed initially; verify existing `TraitBinding` / `ImplBinding` are sufficient.

- `lib/syntax/core_parser.mly`
  - Reuse existing module-level trait/impl syntax.
  - Possibly extend module `let` binding syntax to allow typed `pub let name : Type = value` if stdlib wants annotated public operators.

- `lib/semantic/typecheck/elaborate.ml`
  - Implement `TraitBinding` / `ImplBinding` in `Surface.Struct` elaboration.
  - Preserve `pub`/private visibility for traits and impls.
  - Register public trait metadata/evidence when importing/opening modules.
  - Avoid extending `init_ctx`'s embedded stdlib walker.

- `lib/backend/interp/nbe.ml`
  - Verify any synthetic impl dictionary bindings evaluate correctly and remain inaccessible through ordinary dot access unless intentionally public.

- `test/semantic/test_elaborate.ml`
  - Add module trait/impl visibility tests.
  - Update equality typing tests for `Eq` semantics and explicit stdlib loading.

- `test/backend/test_core.ml`
  - Update runtime equality tests to explicitly source stdlib.
  - Add local helpers for type-case-specific tests that need equality.

## Validation

Run focused tests first:

```sh
dune exec test/syntax/test_syntax.exe
dune exec test/semantic/test_elaborate.exe -- test traits
dune exec test/semantic/test_elaborate.exe -- test operators
dune exec test/backend/test_core.exe -- test eval
```

Then run the full suite:

```sh
dune test
```

Add regression coverage for:

- `pub trait` visible from imported module.
- private trait hidden from imported module.
- public impl evidence available only through explicit import/open.
- private impl evidence not available externally.
- primitive `Eq` equality works after explicit stdlib sourcing.
- unsupported equality fails at elaboration time.
