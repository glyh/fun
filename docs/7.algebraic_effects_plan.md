# Algebraic effects implementation plan

## Goal

Add algebraic effects and handlers to `fun` in a way that fits the current dependent core pipeline:

```text
Surface.t
  -> Elaborate.on_expr
  -> Core.term + semantic type
  -> NbE / interpreter
  -> later VM/CPS backend
```

The first version should prioritize a clear semantic model over performance. Keep syntax minimal until the core/type/runtime model is proven.

## Phase 1: Decide the core effect model

Introduce effects as named operations grouped by effect signatures. Use the same parameterized declaration style as parameterized structs:

```fun
let State S = effect
  get : Unit -> S;
  put : S -> Unit;
end
```

This should behave like a parameterized nominal effect family, not like an arbitrary runtime function. `State` is applied to a type parameter to produce an effect instance:

```fun
State I64
State String
```

Core concepts needed:

- effect declarations
- operation names
- operation parameter type
- operation result type
- effect rows / effect sets on function types
- handlers that interpret operations

Effects should be nominal, like ADTs, even though the declaration syntax resembles parameterized structs. Structural equality is a bad fit for handler dispatch: two independently declared effects with the same operation names and types should still be distinct.

Effect equality should be:

1. same effect family nominal id;
2. equal instantiated parameters.

So:

```fun
State I64 == State I64
State I64 != State String
State I64 != Env I64
```

The declaration should be applicative rather than generative: repeated references to the same family and equal parameters produce the same effect identity. Arbitrary computed effects can wait; the initial implementation should treat `let State S = effect ... end` as a declaration form for a nominal family.

## Phase 2: Extend function types with effects

Current function type is morally:

```fun
A -> B
```

Use `can` for human-readable effect annotations:

```fun
A -> B can E
```

where `E` is an effect row/set.

Examples:

```fun
read_line : Unit -> String can IO
get       : Unit -> S can State S
put       : S -> Unit can State S
pure_add  : I64 -> I64 -> I64
```

Pure functions keep the existing syntax. Multiple effects are comma-separated:

```fun
program : Unit -> I64 can State I64, IO
```

Core changes:

- extend `Core.Pi` / function type representation with an effect annotation;
- extend lambdas/applications to propagate effects;
- update type equality/unification for effect annotations;
- update pretty-printing/debug output.

Initial simplification: use closed effect sets only.

Instead of full row polymorphism immediately:

```fun
{e} -> A -> B can State | e
```

start with:

```fun
A -> B can State, IO
```

Row polymorphism can come later. The internal representation should still be row-shaped enough that open rows can be added without replacing the whole design.

## Phase 3: Add surface syntax for `perform`

Minimal syntax:

```fun
perform State.get ()
perform State.put 42
```

Avoid over-designing syntax at first. The important semantic form is:

```text
Perform(effect_operation, argument)
```

Possible surface AST shape:

```ocaml
| Perform of t * t
```

or, if operation paths are special:

```ocaml
| Perform of path * t
```

Possible core term shape:

```ocaml
| Perform of operation_id * term
```

Typing rule:

If operation has type:

```fun
A -> B
```

then:

```fun
perform op arg
```

has type `B` and contributes effect `{op.effect}`.

## Phase 4: Add effect branches to `match`

Reuse `match` as the handler form. Ordinary branches handle final values; `effect` branches handle suspended operations. There is no explicit `return` marker.

```fun
match expr with
| 0 -> ...
| x -> ...
| effect State.get () k -> ...
| effect State.put s k -> ...
end
```

Most pattern matching does not care about effects, so ordinary matches stay ordinary. Effect branches are only needed when the programmer wants to handle operations.

Effect branches should support the existing pattern language for operation arguments, including tuple and record destructuring:

```fun
match expr with
| x -> x
| effect Console.log (level, message) k -> ...
| effect Ask.prompt Request {text; default} k -> ...
end
```

A handler match needs:

- ordinary value branches, just like existing `match`;
- one effect branch per handled operation;
- access to a continuation `k` in each effect branch.

Important type shape:

If:

```fun
expr : A can E
```

and the match handles `State`, producing `B`, then:

```fun
match expr with ... : B can (E - State + branch_effects)
```

Initial simplifications:

- require handlers to handle a closed known effect set;
- decide whether continuations are one-shot or multi-shot.

Recommendation: start with one-shot continuations semantically, even if represented functionally at first. Multi-shot continuations can be explored later, but one-shot maps better to a future VM.

## Phase 5: Runtime semantics in the interpreter

Before VM/CPS, implement effects in the current evaluator with explicit continuation representation.

Conceptually, evaluation can return either:

```ocaml
type result =
  | Done of value
  | Effect of {
      op : operation_id;
      arg : value;
      k : value -> result;
    }
```

Then:

- normal evaluation returns `Done v`;
- `perform op arg` returns `Effect { op; arg; k }`;
- `match expr with ...` evaluates `expr`;
  - if `Done v`, match `v` against ordinary value branches;
  - if `Effect` is for a handled op, match the operation argument against the effect branch pattern and run the branch with the continuation;
  - if unhandled, bubble the effect outward with a wrapped continuation.

This can initially live in `lib/backend/interp/nbe.ml`, but it will stretch NbE. Long term, effects want a separate evaluator or CPS backend.

## Phase 6: Elaborator changes

Elaboration needs to track both:

```text
term : type
term may perform effects : effect_set
```

So inference/checking becomes something like:

```ocaml
infer : ctx -> Surface.t -> Core.term * value_type * effect_set
check : ctx -> Surface.t -> expected_type -> Core.term * effect_set
```

Rules:

- literals: no effects
- variable lookup: no effects
- function application: effects from function expression, argument expression, and function latent effect
- lambda: body effects become latent effect of function type
- perform: contributes operation's effect
- match with effect branches: ordinary branches handle final values; effect branches remove handled effects and add effects from branch bodies

Initial simplifications:

- no effect polymorphism;
- no subeffecting except set inclusion;
- handlers must explicitly cover operations for the handled effect.

## Phase 7: Tests

Add tests in layers.

### Parser tests

- parse effect declarations
- parse `perform`
- parse `effect` declarations
- parse `can` annotations
- parse `effect` branches in `match`

### Elaboration tests

- pure expressions have empty effect set
- performing an operation adds the correct effect
- pure function cannot call effectful operation unless its type says so
- match effect branches remove handled effects
- unhandled effects remain in the inferred effect set
- wrong operation argument type is rejected
- wrong continuation result type is rejected

### Runtime tests

- simple exception-like effect
- state effect
- ordinary match branches handle final values
- nested handlers
- unhandled effect bubbles outward
- continuation resumes correctly

Example effect test:

```fun
let Exc = effect
  raise : I64 -> I64;
end in

match perform Exc.raise 1 with
| x -> x
| effect Exc.raise n k -> n + 1
end
```

Expected result:

```fun
2
```

State example later:

```fun
match
  let x = perform State.get () in
  perform State.put (x + 1);
  perform State.get ()
with
| x -> x
| effect State.get () k -> ...
| effect State.put s k -> ...
end
```

## Phase 8: Later VM/CPS path

Once interpreter semantics work, lower effectful code through CPS:

```text
Core.term
  -> effect-aware CPS
  -> closure conversion
  -> bytecode
  -> VM
```

Effects become much more natural after CPS because continuations are explicit.

The VM eventually needs:

- continuation frames;
- handler stack;
- operation dispatch;
- captured/resumable continuations;
- one-shot or multi-shot continuation discipline.

## Suggested implementation order

1. Add internal `EffectSet` / `EffectSig` / operation IDs.
2. Add effect annotations to core function types.
3. Thread effect sets through elaboration.
4. Add minimal `perform`.
5. Add runtime bubbling result type: `Done | Effect`.
6. Add `match` effect branches.
7. Add tests for exception-like effects.
8. Add tests for state-like effects.
9. Only then improve syntax.
10. Later: effect polymorphism and VM/CPS lowering.

## Settled initial design choices

- Effects use `let Name params = effect ... end` declaration syntax.
- Parameterized effects are nominal effect families with applicative identity.
- Function effect annotations use `can`.
- Handlers reuse `match`; ordinary branches handle final values and `effect` branches handle operations.
- Effect branch operation arguments use the existing pattern language for destructuring.
- Start with closed effect rows, but keep the internal representation row-shaped.
- Start with one-shot continuations as the future VM-friendly default.
- Include a tiny handler model in the first executable milestone; `perform` alone is not very useful.
