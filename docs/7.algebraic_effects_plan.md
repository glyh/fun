# Algebraic effects implementation plan

## Goal

Add algebraic effects and handlers to `fun` in a way that fits the current dependent core pipeline:

```text
Surface.t -> Elaborate.on_expr -> Core.term + semantic type -> NbE / interpreter
```

The roadmap prioritizes source-level semantics, typechecking, and interpreter behavior. Optimization, VM, and CPS-lowering work are intentionally out of scope for now.

## Phase checklist

- [x] Phase 1: Nominal effect families
- [x] Phase 2: Closed latent effect rows on function types
- [x] Phase 3: `perform` syntax and elaboration
- [x] Phase 4: Match-based effect handlers
- [x] Phase 5: Handler semantics hardening
- [x] Phase 6: Open effect rows / effect polymorphism

## Phase 1: Nominal effect families

Status: implemented.

Effects are nominal families of operations, declared with `effect Name(params) = sig ... end`. Applying an effect family to equal parameters produces the same effect instance.

Design requirements:

- effect identity is nominal, not structural;
- parameterized effects are applicative;
- operation signatures have one input type and one output type;
- operation payload types may include record struct types.

## Phase 2: Closed latent effect rows on function types

Status: implemented.

Function types use `can` to describe latent effects. The initial model uses closed rows, which is enough for concrete programs and handlers before effect polymorphism exists.

Design requirements:

- pure functions keep the existing arrow syntax;
- effectful functions annotate their latent effects with `can`;
- multiple effects are represented as a closed row;
- type equality compares nominal effect instances and parameters.

## Phase 3: `perform` syntax and elaboration

Status: implemented.

`perform Effect.operation arg` invokes an operation and contributes the corresponding effect instance to the expression's effect set.

Design requirements:

- resolve operation paths statically;
- typecheck the operation argument against the declared input type;
- return the declared output type;
- preserve concrete effect instances for parameterized effects;
- reject unhandled effects in pure contexts.

## Phase 4: Match-based effect handlers

Status: implemented.

Handlers reuse `match`. Ordinary branches handle final values; `effect` branches handle suspended operations.

Design requirements:

- effect branches use the existing pattern language for operation arguments;
- handler dispatch compares nominal effect instances, not just operation names;
- continuations are exposed through `resume arg`, not a user-named binder;
- `resume` is only valid inside effect branch bodies;
- continuations are one-shot.

## Phase 5: Handler semantics hardening

Status: implemented for closed rows.

The current handler model is now specified and covered by focused parser, elaboration, and backend tests before expanding the type system.

Completed work:

- [x] specified handlers as deep for branch bodies and resumed continuations;
- [x] ensured branch bodies run under the active handler loop;
- [x] tested ordinary value branches with effect branches present;
- [x] tested unhandled effects bubbling through inner matches;
- [x] tested multiple effects ping-ponging through handlers;
- [x] tested fully handled multi-operation effects;
- [x] tested partial multi-operation handlers preserving residual effects;
- [x] rejected duplicate operation branches for the same full effect instance;
- [x] distinguished handler branches for different parameter instantiations of the same effect family;
- [x] rejected wrong operation argument patterns;
- [x] rejected wrong effect branch result types;
- [x] tested tuple and record operation payload patterns and binding order;
- [x] rejected `resume` outside effect branches;
- [x] required `resume` to receive an argument;
- [x] allowed lexical `resume` inside nested lambdas in an effect branch;
- [x] kept runtime continuation reuse as a clear one-shot error;
- [x] kept direct unhandled `perform` errors clear.

Notes:

- `resume` remains a keyword-backed syntax form, not a normal variable.
- `resume` is lexically scoped through nested lambdas in an effect branch, but continuations remain one-shot at runtime.
- Handler branch identity is the full nominal effect instance plus operation name. `State I64.get` and `State Bool.get` are distinct branches.
- Open row tails are now implemented; handler semantics preserve residual tails when concrete effects are removed.

## Phase 6: Open effect rows / effect polymorphism

Status: implemented.

Higher-order functions can preserve unknown effects from callbacks and arguments instead of forcing every effect row to be closed. Omitted effect annotations on function types infer/thread a row tail by default, while `can {}` is the explicit pure-row annotation.

Implemented work:

- [x] chose surface syntax for row-polymorphic effect annotations: `can {IO | r}`, `can {IO, State I64 | r}`, and `can {| r}`;
- [x] added explicit pure syntax with `can {}`;
- [x] represented closed rows and open rows with row tails;
- [x] added first-class `EffectRow` and semantic row literals;
- [x] introduced row metavariables during omitted-effect inference;
- [x] extended NbE, quotation, conversion, and unification for row variables;
- [x] normalized row tails and kept concrete effects order-insensitive/set-like;
- [x] preserved nominal effect-instance equality in rows;
- [x] propagated row variables through lambda and application inference;
- [x] computed handler residual rows for open rows;
- [x] preserved unrelated effects and row tails when a handler removes concrete effects;
- [x] updated diagnostics-adjacent debug printing for open rows;
- [x] tested explicit pure annotations, higher-order callback effect threading, open-row annotations, duplicate rows, and row-unification failures.

Notes:

- Regular `A -> B` no longer means pure; it infers a row tail. Use `A -> B can {}` when purity is required.
- Explicit `can IO` and `can {IO, State I64}` remain closed rows.
- Open rows can name a concrete prefix plus tail or be tail-only.
- The current implementation is still prototype-level and intentionally avoids full lacks-constraint machinery.

## Settled design choices

- Effects use `effect Name(params) = sig ... end` declaration syntax.
- Parameterized effects are nominal effect families with applicative identity.
- Function effect annotations use `can`.
- Handlers reuse `match`.
- Effect branch arguments use normal pattern syntax.
- Handler continuations use `resume arg`.
- Handlers are deep for branch bodies and resumed continuations.
- `resume` is lexically available inside nested lambdas in an effect branch.
- Continuations are one-shot.
- Handler branch identity includes the full parameterized effect instance and operation name.
- Omitted effect annotations infer/thread row tails by default; `can {}` is the explicit pure row.
- Closed and open rows coexist: `can IO`, `can {IO, State I64}`, `can {IO | r}`, and `can {| r}`.
- Optimization, VM, and CPS lowering are out of scope for this plan.
