# Implement mutable references

## Context

`ROADMAP.md` lists References as the next language feature after the recent regression coverage work. The only explicit semantic requirement is that mutable ref cells must capture the cell, not the current value, inside closures. The goal is to add a small first-class mutable reference feature while preserving the dependent typechecker/NbE invariants as much as possible.

Recommended first scope:

- `Ref(A) : Type`
- `ref(e) : Ref(A)`
- `deref(r) : A`
- `r <- e : Unit`
- no user-visible effect-row tracking for refs in this first pass
- runtime refs are opaque mutable cells; closures and aliases share the same cell object

## Implementation plan

1. Add surface syntax.
   - Update `lib/syntax/surface.ml` with `RefNew`, `RefGet`, and `RefSet` expression variants.
   - Update recursive surface traversals in `lib/semantic/typecheck/elaborate.ml`, especially `rewrite_record_self_refs`, to recurse through those variants.

2. Extend reader/enforestation.
   - In `lib/syntax/raw_syntax.ml` / `lib/expand/enforest.ml`, recognize `ref`, `deref`, and assignment.
   - Parse:
     - `ref(e)` as `RefNew e`
     - `deref(e)` as `RefGet e`
     - `lhs <- rhs` as `RefSet (lhs, rhs)`
   - Keep precedence conservative: assignment low-precedence; prefix ref/deref near existing `perform`/`resume` forms. Require parentheses for compound prefix arguments where needed.

3. Add core/runtime representations.
   - In `lib/core_kernel/core.ml`, add core terms:
     - `RefTy of term`
     - `RefNew of term`
     - `RefGet of term`
     - `RefSet of term * term`
   - Add semantic values:
     - `VRefTy of value`
     - `VRef of value ref`
   - Add neutral frames for open-term robustness:
     - `FRefGet`
     - `FRefSet of value`
    - This keeps `fn(r) -> deref(r)` and similar open terms representable during NbE rather than forcing an evaluator error on rigid variables.

4. Add evaluator support.
   - In `lib/backend/interp/nbe.ml`:
     - `RefTy a` evaluates to `VRefTy (eval a)`.
     - `RefNew e` evaluates `e` and allocates `VRef (ref value)`.
     - `RefGet r` reads `VRef` cells, or creates a neutral `FRefGet` for stuck refs.
     - `RefSet (r, e)` writes `VRef` cells and returns `VAtom Unit`, or creates a neutral `FRefSet value` for stuck refs.
   - Update `quote`:
     - quote `VRefTy` structurally to `RefTy`.
     - quote neutral ref frames back to `RefGet`/`RefSet` terms.
     - reject quoting live `VRef _`, like `VCont _`.
   - Update `conv` structurally for `VRefTy` and neutral ref frames.
   - Do not compare `VRef` by current contents; keep live cells opaque.

5. Add typechecking/elaboration.
   - Expose `Ref` in `init_ctx` in `lib/semantic/typecheck/elaborate.ml` as a builtin type constructor equivalent to `fn(A) -> RefTy(A)` with type `Type -> Type`. This keeps `Ref(I64)` as ordinary application syntax.
   - Add inference rules:
      - `ref(e)`: infer `e : A`, return `RefNew e_core : Ref(A)`.
      - `deref(r)`: infer/refine `r : Ref(A)`, return `RefGet r_core : A`.
       - `r <- e`: infer/refine `r : Ref(A)`, check `e : A`, return `RefSet (r_core, e_core) : Unit`.
   - Add `VRefTy` to type-like handling; keep `VRef` non-type-like.
   - Update `collect_effects` to recurse through ref expressions but not add user-visible effects.

6. Prevent mutation during typechecking.
   - The elaborator currently evaluates pure let-bound RHSs into the context with `Ctx.define` when `collect_effects` is empty. Since refs are intentionally not effect-row tracked, add an internal syntactic/value restriction helper such as `compile_time_safe`.
   - Mark `RefNew`, `RefGet`, and `RefSet` as not compile-time-safe.
   - In `collect_effects` let handling, `infer_let`, and `check` let handling, only use `Ctx.define` when both:
     - the expression has no collected effects, and
     - `compile_time_safe value` is true.
   - Otherwise use `Ctx.bind`, preserving the type without executing ref allocation/read/write during elaboration.
    - Restrict mutable ref operations from type expressions: `Ref(A)` is allowed in types, but `ref(e)`, `deref(r)`, and `r <- e` should not be accepted as type-level computation.

7. Update unification and debug support.
   - In `lib/semantic/typecheck/unify.ml`:
     - structurally rename/unify `VRefTy`.
     - reject renaming/unifying live `VRef _`, like continuations.
     - traverse `VRefTy` in occurs checks.
     - add `value_form` names for reference types/cells.
     - support neutral `FRefGet`/`FRefSet` frame comparison.
   - In `lib/core_kernel/debug.ml`, add concise printers for ref terms, `VRefTy`, and opaque `VRef _` (`<ref>`).

8. Add regression tests.
   - Syntax tests in `test/syntax/`:
      - `ref(1)`
      - `deref(r)`
      - `r <- 2`
      - `r <- deref(r) + 1`
     - `1 != 2` still parses as inequality.
   - Semantic tests in `test/semantic/test_elaborate.ml`:
      - `Ref(I64)` has type `Type`.
      - `(ref(1) : Ref(I64))` elaborates.
      - `do r = ref(1); deref(r) end` has type `I64`.
      - `do r = ref(1); r <- 2 end` has type `Unit`.
     - wrong assignment type is rejected.
     - deref/assignment on non-ref is rejected.
   - Backend tests in `test/backend/test_core.ml`:
     - allocation/read returns initial value.
     - write/read observes the new value.
     - aliases share the same cell.
     - closures observe later writes to the same cell.
     - repeated closure increments mutate the same captured cell.

## Verification

Run:

```sh
dune test
```

Also manually sanity-check the REPL or backend tests for the roadmap invariant:

```fun
do
  r = ref(0)
  f = fn(_) -> deref(r)
  _ = r <- 4
  f()
end
```

Expected result: `4`, proving the closure captured the cell rather than the old value.

## Scope notes / risks

- Do not implement reference equality or structural equality over cells in this pass.
- Do not make mutable operations part of effect rows yet; use the internal compile-time safety check to protect elaboration instead.
- Live `VRef` cells should remain runtime-only and unquotable to avoid embedding mutable state into normal forms, metas, or definitional equality.
- The future closure-conversion implementation must preserve this same heap-cell identity behavior; backend alias/closure tests should guard that invariant.
